39  /**
40   * <h1>SNACKRequestParser</h1>
41   * <p>
42   * This class parses a SNACK Request message defined in the iSCSI Standard (RFC3720).
43   * <p>
44   * If the implementation supports ErrorRecoveryLevel greater than zero, it MUST support all SNACK types.
45   * <p>
46   * The SNACK is used by the initiator to request the retransmission of numbered-responses, data, or R2T PDUs
47   * from the target. The SNACK request indicates the numbered-responses or data "runs" whose retransmission is
48   * requested by the target, where the run starts with the first StatSN, DataSN, or R2TSN whose retransmission
49   * is requested and indicates the number of Status, Data, or R2T PDUs requested including the first.
50   * <code>0</code> has special meaning when used as a starting number and length:
51   * <p>
52   * <ul>
53   * <li>When used in RunLength, it means all PDUs starting with the initial.</li>
54   * <li>When used in both BegRun and RunLength, it means all unacknowledged PDUs.</li>
55   * </ul>
56   * The numbered-response(s) or R2T(s), requested by a SNACK, MUST be delivered as exact replicas of the ones
57   * that the target transmitted originally except for the fields ExpCmdSN, MaxCmdSN, and ExpDataSN, which MUST
58   * carry the current values. R2T(s)requested by SNACK MUST also carry the current value of StatSN.
59   * <p>
60   * The numbered Data-In PDUs, requested by a Data SNACK MUST be delivered as exact replicas of the ones that
61   * the target transmitted originally except for the fields ExpCmdSN and MaxCmdSN, which MUST carry the current
62   * values and except for resegmentation (see Section 10.16.3 Resegmentation).
63   * <p>
64   * Any SNACK that requests a numbered-response, Data, or R2T that was not sent by the target or was already
65   * acknowledged by the initiator, MUST be rejected with a reason code of "Protocol error".
66   * <p>
67   * <h4>Data Acknowledgement</h4> If an initiator operates at ErrorRecoveryLevel <code>1</code> or higher, it
68   * MUST issue a SNACK of type DataACK after receiving a Data-In PDU with the A bit set to <code>1</code>.
69   * However, if the initiator has detected holes in the input sequence, it MUST postpone issuing the SNACK of
70   * type DataACK until the holes are filled. An initiator MAY ignore the A bit if it deems that the bit is
71   * being set aggressively by the target (i.e., before the MaxBurstLength limit is reached).
72   * <p>
73   * The DataACK is used to free resources at the target and not to request or imply data retransmission.
74   * <p>
75   * An initiator MUST NOT request retransmission for any data it had already acknowledged.
76   * <p>
77   * <h4>Resegmentation</h4> If the initiator MaxRecvDataSegmentLength changed between the original transmission
78   * and the time the initiator requests retransmission, the initiator MUST issue a R-Data SNACK (see Section
79   * 10.16.1 Type). With R-Data SNACK, the initiator indicates that it discards all the unacknowledged data and
80   * expects the target to resend it. It also expects resegmentation. In this case, the retransmitted Data-In
81   * PDUs MAY be different from the ones originally sent in order to reflect changes in
82   * MaxRecvDataSegmentLength. Their DataSN starts with the BegRun of the last DataACK received by the target if
83   * any was received; otherwise it starts with 0 and is increased by 1 for each resent Data-In PDU.
84   * <p>
85   * A target that has received a R-Data SNACK MUST return a SCSI Response that contains a copy of the SNACK Tag
86   * field from the R-Data SNACK in the SCSI Response SNACK Tag field as its last or only Response. For example,
87   * if it has already sent a response containing another value in the SNACK Tag field or had the status
88   * included in the last Data-In PDU, it must send a new SCSI Response PDU. If a target sends more than one
89   * SCSI Response PDU due to this rule, all SCSI responses must carry the same StatSN (see Section 10.4.4 SNACK
90   * Tag). If an initiator attempts to recover a lost SCSI Response (with a Status SNACK, see Section 10.16.1
91   * Type) when more than one response has been sent, the target will send the SCSI Response with the latest
92   * content known to the target, including the last SNACK Tag for the command.
93   * <p>
94   * For considerations in allegiance reassignment of a task to a connection with a different
95   * MaxRecvDataSegmentLength, refer to Section 6.2.2 Allegiance Reassignment.
96   * <p>
97   * <h4>Initiator Task Tag</h4> For Status SNACK and DataACK, the Initiator Task Tag MUST be set to the
98   * reserved value <code>0xffffffff</code>. In all other cases, the Initiator Task Tag field MUST be set to the
99   * Initiator Task Tag of the referenced command.
100  * <p>
101  * 
102  * @author Volker Wildi
103  */
