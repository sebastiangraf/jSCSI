37      /**
38       * Use: During Login - Security Negotiation Senders: Initiator and Target
39       * Scope: connection AuthMethod = &lt;list-of-values&gt; The main item of
40       * security negotiation is the authentication method (AuthMethod). The
41       * authentication methods that can be used (appear in the list-of-values)
42       * are either those listed in the following table or are vendor-unique
43       * methods:
44       * <p/>
45       * <table border="1">
46       * <tr>
47       * <th>Name</th>
48       * <th>Description</th>
49       * </tr>
50       * <tr>
51       * <td>KRB5</td>
52       * <td>Kerberos V5 - defined in [RFC1510]</td>
53       * </tr>
54       * <tr>
55       * <td>SPKM1</td>
56       * <td>Simple Public-Key GSS-API Mechanism defined in [RFC2025]</td>
57       * </tr>
58       * <tr>
59       * <td>SPKM2</td>
60       * <td>Simple Public-Key GSS-API Mechanism defined in [RFC2025]</td>
61       * </tr>
62       * <tr>
63       * <td>SRP</td>
64       * <td>Secure Remote Password defined in [RFC2945]</td>
65       * </tr>
66       * <tr>
67       * <td>CHAP</td>
68       * <td>Challenge Handshake Authentication Protocol defined in [RFC1994]</td>
69       * </tr>
70       * <tr>
71       * <td>None</td>
72       * <td>No authentication</td>
73       * </tr>
74       * </table>
75       * <br/>
76       * The AuthMethod selection is followed by an "authentication exchange" specific to the authentication
77       * method selected. The authentication method proposal may be made by either the initiator or the target.
78       * However the initiator MUST make the first step specific to the selected authentication method as soon
79       * as it is selected. It follows that if the target makes the authentication method proposal the initiator
80       * sends the first keys(s) of the exchange together with its authentication method selection. The
81       * authentication exchange authenticates the initiator to the target, and optionally, the target to the
82       * initiator. Authentication is OPTIONAL to use but MUST be supported by the target and initiator. The
83       * initiator and target MUST implement CHAP. All other authentication methods are OPTIONAL. Private or
84       * public extension algorithms MAY also be negotiated for authentication methods. Whenever a private or
85       * public extension algorithm is part of the default offer (the offer made in absence of explicit
86       * administrative action) the implementer MUST ensure that CHAP is listed as an alternative in the default
87       * offer and "None" is not part of the default offer. Extension authentication methods MUST be named using
88       * one of the following two formats:
89       * <ol>
90       * <li>Z-reversed.vendor.dns_name.do_something=</li>
91       * <li>Z&lt;#&gt;&lt;IANA-registered-string&gt;=</li>
92       * </ol>
93       * Authentication methods named using the Z- format are used as private extensions. Authentication methods
94       * named using the Z# format are used as public extensions that must be registered with IANA and MUST be
95       * described by an informational RFC. For all of the public or private extension authentication methods,
96       * the method specific keys MUST conform to the format specified in Section 5.1 Text Format for
97       * standard-label. To identify the vendor for private extension authentication methods, we suggest you use
98       * the reversed DNS-name as a prefix to the proper digest names. The part of digest-name following Z- and
99       * Z# MUST conform to the format for standard-label specified in Section 5.1 Text Format. Support for
100      * public or private extension authentication methods is OPTIONAL. The following subsections define the
101      * specific exchanges for each of the standardized authentication methods. As mentioned earlier the first
102      * step is always done by the initiator.
103      */
104     AUTH_METHOD("AuthMethod"),

106     /**
107      * Use: IO <br/>
108      * Senders: Initiator and Target<br/>
109      * Scope: CO
110      * <p/>
111      * HeaderDigest = &lt;list-of-values&gt; <br/>
112      * DataDigest = &lt;list-of-values&gt;
113      * <p/>
114      * Default is None for both HeaderDigest and DataDigest.
115      * <p/>
116      * Digests enable the checking of end-to-end, non-cryptographic data integrity beyond the integrity checks
117      * provided by the link layers and the covering of the whole communication path including all elements
118      * that may change the network level PDUs such as routers, switches, and proxies. <br/>
119      * The following table lists cyclic integrity checksums that can be negotiated for the digests and that
120      * MUST be implemented by every iSCSI initiator and target. These digest options only have error detection
121      * significance.
122      * <p/>
123      * <table border="1">
124      * <tr>
125      * <th>Name</th>
126      * <th>Description</th>
127      * <th>Generator</th>
128      * </tr>
129      * <tr>
130      * <td>CRC32CDigest</td>
131      * <td>32 bit CRC</td>
132      * <td>0x11edc6f41</td>
133      * </tr>
134      * <tr>
135      * <td>None</td>
136      * <td colspan="2">no digest</td>
137      * </tr>
138      * </table>
139      * <br/>
140      * The generator polynomial for this digest is given in hex-notation (e.g., 0x3b stands for 0011 1011 and
141      * the polynomial is x**5+X**4+x**3+x+1).
142      * <p/>
143      * When the Initiator and Target agree on a digest, this digest MUST be used for every PDU in Full Feature
144      * Phase. <br/>
145      * Padding bytes, when present in a segment covered by a CRC, SHOULD be set to 0 and are included in the
146      * CRC. <br/>
147      * The CRC MUST be calculated by a method that produces the same results as the following process: <br/>
148      * <ul>
149      * <li>The PDU bits are considered as the coefficients of a polynomial M(x) of degree n-1; bit 7 of the
150      * lowest numbered byte is considered the most significant bit (x^n-1), followed by bit 6 of the lowest
151      * numbered byte through bit 0 of the highest numbered byte (x^0).</li>
152      * <li>The most significant 32 bits are complemented.</li>
153      * <li>The polynomial is multiplied by x^32 then divided by G(x). The generator polynomial produces a
154      * remainder R(x) of degree <= 31.</li>
155      * <li>The coefficients of R(x) are considered a 32 bit sequence.</li>
156      * <li>The bit sequence is complemented and the result is the CRC.</li>
157      * <li>The CRC bits are mapped into the digest word. The x^31 coefficient in bit 7 of the lowest numbered
158      * byte of the digest continuing through to the byte up to the x^24 coefficient in bit 0 of the lowest
159      * numbered byte, continuing with the x^23 coefficient in bit 7 of next byte through x^0 in bit 0 of the
160      * highest numbered byte.</li>
161      * <li>Computing the CRC over any segment (data or header) extended to include the CRC built using the
162      * generator 0x11edc6f41 will always get the value 0x1c2d19ed as its final remainder (R(x)). This value is
163      * given here in its polynomial form (i.e., not mapped as the digest word).</li>
164      * </ul>
165      * For a discussion about selection criteria for the CRC, see [RFC3385]. For a detailed analysis of the
166      * iSCSI polynomial, see [Castagnoli93]. <br/>
167      * Private or public extension algorithms MAY also be negotiated for digests. Whenever a private or public
168      * digest extension algorithm is part of the default offer (the offer made in absence of explicit
169      * administrative action) the implementer MUST ensure that CRC32CDigest is listed as an alternative in the
170      * default offer and "None" is not part of the default offer. <br/>
171      * Extension digest algorithms MUST be named using one of the following two formats: <br/>
172      * <ol>
173      * <li>Y-reversed.vendor.dns_name.do_something=</li>
174      * <li>Y&lt;#&gt;&lt;IANA-registered-string&gt;=</li>
175      * </ol>
176      * Digests named using the Y- format are used for private purposes (unregistered). Digests named using the
177      * Y# format (public extension) must be registered with IANA and MUST be described by an informational
178      * RFC. <br/>
179      * For private extension digests, to identify the vendor, we suggest you use the reversed DNS-name as a
180      * prefix to the proper digest names. <br/>
181      * The part of digest-name following Y- and Y# MUST conform to the format for standard-label specified in
182      * Section 5.1 Text Format. Support for public or private extension digests is OPTIONAL.
183      */
184     HEADER_DIGEST("HeaderDigest"),
